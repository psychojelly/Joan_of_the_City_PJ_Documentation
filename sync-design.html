Audio Sync Design WIP

A proposed three-layer design to bring every device's audio into sync — shared clock, scheduled starts, and a position servo — riding the existing OSC path. Not yet implemented; pending review by the Lead Developer.

🚧

Status: proposal. The full implementation handoff (message formats, per-phase steps for Unity / the servers / the tablet app, acceptance tests, and ready-to-use Claude Code prompts) lives with the dev team as AUDIO-SYNC-HANDOFF.md in the cue-controller app repo. This page is the summary of record.

Sync

The problem

Today a cue means “play the moment the packet happens to arrive.” Every source of arrival wobble becomes an audio wobble.

  • Wi-Fi delivery varies per device (jitter, retries)
  • Each device's audio pipeline takes a different time to start
  • A stem not yet preloaded loads at cue time — the big one
  • Once started, device clocks drift apart over long stems

Measured side by side: devices land 25 ms to 1 s apart. The ~25 ms floor is jitter + pipeline variance; the 1 s outliers are loading/stalls at cue time. The sync budget is set by the live singer, who performs in real air against what every listener hears: target ±10–20 ms.

Sync

The design — three layers

Built in order; each is independently shippable and fully backwards-compatible (a cue without a timestamp behaves exactly as today).

Shared clock

Every device pings the master (the operator cue device) NTP-style — offset ≈ ts − (t0+t1)/2, filtered to the lowest-RTT samples — agreeing on “master time” to ~1–5 ms. Runs continuously, so it also absorbs crystal drift and survives Wi-Fi dropouts (offsets don't jump when the link does).

Scheduled start

Cues carry a future timestamp: /audio/cue "B_201" playAt=masterTime+400ms. Each device schedules the start on its audio thread (Unity AudioSource.PlayScheduled) — so any packet arriving inside the lead window fires at the identical instant, immune to network jitter and main-thread hitches. Late arrival? Start immediately but seek in by the overshoot, landing mid-phrase in sync. 25 ms floor → ~5 ms.

Position servo

The master broadcasts /sync stem, position, masterTime 1–2×/sec. Devices compare their playback position to the age-adjusted expectation: small error → inaudible rate nudge; big error → seek; stem not playing but should be → start it at that position. This corrects long-stem drift and gives reboot / missed-cue / late-join recovery for free.

Delivery, mobility & multicast

Design decisions at the network layer, and why.

  • Stay unicast; no multicast. Wi-Fi unicast frames get link-layer ACKs and automatic retries; multicast frames get none (sent once, blind, at a low basic rate — and Android filters multicast in firmware by default). This holds under 802.11ax too, where OFDMA effectively parallelizes the per-device sends anyway.
  • Redundant sends as loss insurance: each scheduled cue is sent 3× at 50 ms spacing; receivers dedupe by (cueId, playAt), which the timestamp makes safe. (Today's visual toggle cues are not idempotent — never blind-resend those.)
  • Moving headsets: movement changes delay (absorbed by the lead window — size it for the worst route leg, 0.5–1 s) but not the clock offset (a property of the clocks, maintained by the continuous filtered sync). If coverage itself is the issue, the network should travel with the group — a carried Wi-Fi 6 travel router per Joan.
