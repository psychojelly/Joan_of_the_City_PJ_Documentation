Sync

The problem

Today a cue means “play the moment the packet happens to arrive.” Every source of arrival wobble becomes an audio wobble.

  • Wi-Fi delivery varies per device (jitter, retries)
  • Each device's audio pipeline takes a different time to start
  • A stem not yet preloaded loads at cue time — the big one
  • Once started, device clocks drift apart over long stems

Measured side by side: devices land 25 ms to 1 s apart. The ~25 ms floor is jitter + pipeline variance; the 1 s outliers are loading/stalls at cue time. The sync budget is set by the live singer, who performs in real air against what every listener hears: target ±10–20 ms.

Sync

The design — three layers

Built in order; each is independently shippable and fully backwards-compatible (a cue without a timestamp behaves exactly as today).

Shared clock

Every device pings the master (the operator cue device) NTP-style — offset ≈ ts − (t0+t1)/2, filtered to the lowest-RTT samples — agreeing on “master time” to ~1–5 ms. Runs continuously, so it also absorbs crystal drift and survives Wi-Fi dropouts (offsets don't jump when the link does).

Scheduled start

Cues carry a future timestamp — “play this cue at master time T”, sent ~400 ms ahead. Each device schedules the start on its audio thread (Unity AudioSource.PlayScheduled) — so any packet arriving inside the lead window fires at the identical instant, immune to network jitter and main-thread hitches. Late arrival? Start immediately but seek in by the overshoot, landing mid-phrase in sync. 25 ms floor → ~5 ms.

Position servo rejected as designed

The master broadcasts each stem's expected position; devices self-correct. Rejected for this production (2026-07-15): the stems are short — every cue start already re-anchors all devices sample-accurately, so drift never accumulates — and rate-nudging pitch-shifts sustained sung material audibly. An optional recovery-only variant (a silent device restarts its stem at the right position; playing audio is never touched) stays on the table pending real-headset cue-miss data.

Sync

Delivery, mobility & multicast

Design decisions at the network layer, and why.

  • Stay unicast; no multicast. Wi-Fi unicast frames get link-layer ACKs and automatic retries; multicast frames get none (sent once, blind, at a low basic rate — and Android filters multicast in firmware by default). This holds under 802.11ax too, where OFDMA effectively parallelizes the per-device sends anyway.
  • Redundant sends as loss insurance: each scheduled cue is sent 3× at 50 ms spacing; receivers dedupe by (cueId, playAt), which the timestamp makes safe. (Today's visual toggle cues are not idempotent — never blind-resend those.)
  • Moving headsets: movement changes delay (absorbed by the lead window — size it for the worst route leg, 0.5–1 s) but not the clock offset (a property of the clocks, maintained by the continuous filtered sync). If coverage itself is the issue, the network should travel with the group — a carried Wi-Fi 6 travel router per Joan.

Debug

Debug mode — the return path

Until now data flowed one way (controller → devices) and nothing reported back. Debug mode closes the loop: devices report to the cue server on a second port, and the operator page visualizes it live.

Devices ──debug reports──▶ cue server (listener)
                              │ buffers + orders
Operator page ◀───── live feed ───── server

All of it is opt-in and OFF by default — until an operator flips the toggle there are no log hooks and no verbose traffic. One deliberate exception: a low-rate safety heartbeat (one small packet per 5 s per device once a master is known) keeps the roster answering “is headset 3 alive?” even mid-show with everything off — untick Always On Safety Heartbeat on the receiver for strictly-zero traffic. Timestamps are master-clock seconds (the same clock the sync uses), so the controller computes true send→receive margins per device per cue.

What flows

Operators broadcast small control toggles (reporting on/off, heartbeat on/off, HUD, test tone, mute, CSV reload); devices reply with an announce, a per-cue receipt with its scheduling margin, a heartbeat carrying clock health and playback state, and rate-limited log lines. The exact message contracts (addresses, argument schemas, ports) are deliberately not published here — they live with the dev team in DEBUG-MODE.md in the cue-controller repo, the source of truth for implementers.

The three surfaces

Operator page — 🐞 panel

Toggles for reporting and heartbeat; a device roster (green/yellow/red liveness dot, clock health — clock ✓ Δ0.2ms offset stability or clock ⚠ unsynced — last cue, last-seen age); a CUE ACKS rollup per cue — ✔ cue A_SQ201 → 3/3 received (401/399/404ms), amber until every device reports; a scrolling, color-coded message log with a second delay-graph view (line chart of each device's heartbeat transit delay over the last 90 s, per-device checkboxes to select series, and the log beneath filtered to the selected devices); and the SYNC TEST row (test tone + mute). Every line is stamped in master-clock seconds — sent and reply on one scale: each control message logs a SENT line at the server's send time (and playAt for scheduled sends), then each device's RX reply lands a few ms later, directly comparable.

Unity headsets — HUD

DebugReporter sends the reports; DebugHud draws an in-glasses overlay — device id, master IP, clock sync state + offset, heartbeat status, and the recent cue/log feed with margins. Exists only while debug is on; no Canvas, prefab, or scene changes.

Performer tablets

Already had a status board; now also report back with the same conventions, so headsets and tablets share one roster.

Sync test utilities

  • 🔊 Test tone: a procedurally generated triple beep (no audio asset) played outside the cue system — no cue CSV, no stems, no transport state. In sync mode it is scheduled on the master clock exactly like a real cue, so firing it at every device turns your ears into the sync test: one clean beep-beep-beep = in sync; a flam = trouble. Deduped against the 3× sends; shows in the logger as cue TEST-TONE with its margin.
  • 🔇 Mute / unmute: silences every targeted device via AudioListener.volume — playback continues underneath, so unmuting drops straight back into sync (pausing would drift).

Safety by construction

  • Default OFF, broadcast to enable — zero debug traffic during a show unless asked for.
  • Heartbeat is its own toggle — roster without log spam.
  • Reuses the MasterClock — no second clock; offsets already meaningful.
  • Rate-limited and reentrancy-guarded — a log storm can't flood the Wi-Fi or recurse.
  • Same rollback story as sync: toggle off = the code paths of today.
