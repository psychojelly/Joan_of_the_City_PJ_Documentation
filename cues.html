Cues

Overview

A cue is a single directed moment in a scene — a light builds, a particle reveal triggers, the raven flock lands, the Shadow stands. The cue sheet is the master list: for each cue it records the asset playing, the AR/visual imagery, the audio, and what the performer is doing.

What a cue captures

FieldMeaning
SceneWhich scene the cue belongs to — Light, Voice, Shadow, Communion, etc.
CueThe directed moment itself, in order within the scene.
AssetWhat is being played — the model, particle system, skin, or effect.
AR / Visual imageryWhat the audience sees in AR at this cue (from the script's imagery column + Milanote).
AudioThe stem or sound playing, from the audio cue sheet.
Performer actionWhat the live performer is doing during the cue.
STOP ALLThe master safety — halts and resets every running cue at once.

Editing & Cueing System

The cue sheet is the single document that ties the show together. It's assembled from the script (the AR / visual imagery column), the scene asset list, the audio cue sheet, and the Milanote reference boards — one row per cue, in performance order. It's the human plan; the technical firing happens over OSC — see The OSC control system below.

The cue sheet

Each row is a cue. Keep the row itself short and scannable; put longer direction in a notes field someone can click into for more detail (so the sheet doesn't sprawl). The current working copy lives in a shared spreadsheet — see Tools & links.

Editing workflow

Pull from the sources

Start from the script's AR/visual imagery column, the scene's asset list, the audio cue sheet, and the Milanote boards.

One row per cue

Record asset, AR/visual, audio, and performer action. Keep the row short; push detail into the notes field.

Add performer action

Every cue should say what the live performer is doing — it's part of the score, not an afterthought.

Verify against the build

Step through the cues in the sandbox or on-headset (see Running a Local Test) and confirm each fires as written — and consistently across devices.

STOP ALL & running the show

At performance time an operator drives the cues from Show-Control. The one control that matters above all is STOP ALL — it halts and resets every running cue instantly. Verify it on the current build before every show.

🔒

STOP ALL must be confirmed working before doors open. If it isn't verified on the build you're running, the show is not ready — treat it as a release blocker.

The OSC control system

At performance time, cues are fired over OSC (Open Sound Control). The operator drives a small web page of buttons; each press is relayed to the show as an OSC message — including STOP ALL.

How a button becomes a cue

Operator clicks a button

In the QLab-style controller — on the Joan Cues tablet app or in a browser served by the PC cue server. One button per cue, plus STOP ALL.

The page calls the local cue server

The page posts the cue to the local server on :8765 — built into the tablet app, or server.py on a PC.

The server sends OSC

It forwards the cue as an OSC/UDP message to every IP in the selected audience (each on port 7000).

The show fires the cue

The OSC target — the Unity show — receives the message and runs the cue.

The pieces

ComponentRole
The controller pageThe operator's QLab-style front end (cue lists, audiences/IP groups, settings). Runs in the tablet app or any browser.
The cue serverBridges the page to OSC/UDP — built into the Joan Cues app, or server.py on a PC (listens on :8765, also answers clock-sync pings on :9001).
OSC targetThe Unity show on the brain PC — an extOSC receiver on the in-scene OSC Manager, UDP port 7000 (host IP hardcoded to 10.0.1.32, update per venue).

Canonical controller + server files live in the app repo under pc-server/ (psychojelly/joan-cue-controller-android); the copy at Assets/ThirdParty/OSC_Controller/ in the Unity project is the original and will drift until the updated files are folded back in.

Audio cues — /audio/*

Audio runs on a separate stem engine (JoanAudio) with its own OSC addresses, so music can be cued independently of the visuals.

AddressAction
/audio/cueTrigger an audio cue.
/audio/jump · /audio/seekJump or seek within a stem.
/audio/volSet volume.
StopAll“stop all” / “kill all” — fades every active stem to 0. This is the audio-side STOP ALL.
⚙️

Setup and run steps (Python, python-osc, starting the proxy) live in System Set Up → Running the OSC controller.

🔌

The buttons do nothing if the cue server isn't running. Before a show, confirm it shows listening on :8765 (on the tablet app: the “Joan cue server running” notification) — and that STOP ALL reaches the targets.

The Joan Cues tablet app

A native Android app that puts the whole cue system on a tablet — the show-grade replacement for running proxy.py on a PC. One APK, two roles, chosen at first launch and remembered.

ModeWhat it does
🎛 OperatorHosts the QLab-style controller (the same webpage, unchanged) on an embedded server and fires OSC cues to every target IP — with wake + Wi-Fi locks so Android can't kill it mid-show. Other devices can open http://<tablet-ip>:8765/ as thin clients.
🎭 PerformerListens on :7000 exactly like a headset and plays the show audio — a Kotlin port of the JoanAudio engine (CSV cues, stem download + cache, Loop/OneShot/Stop/StopAll, volume ramps) with a glanceable status board: current cue, active stems, log.
  • Role memory: the choice is remembered per tablet; switching requires a confirmed tap (SWITCH MODE in Performer, Back in Operator) so a mis-tap can't change a role mid-show.
  • Per-Joan setup: configure a Joan's IP list/audiences in the controller, Export Settings to a .json, Import on that Joan's tablet — the settings workflow works fully on-device.
  • Audio sync: the app implements sync phases 0+1 as a test version — Operator stamps scheduled cues (toggleable, off = old behavior), Performer clock-syncs and schedules starts. Validated end-to-end on the emulator (clock offset 0.1 ms; cue scheduled at 1498/1500 ms; 3× dedupe working).
  • Source & docs: private repo psychojelly/joan-cue-controller-android — its README covers dev setup, build, sideload, and the test plan; AUDIO-SYNC-HANDOFF.md there is the sync implementation handoff.
📱

Install model: sideloaded APK, same as the headsets — no Play Store. Runs on Android 7.0+ (incl. Fire HD 8 2018+, with the Fire's weaker Wi-Fi noted for show use).

Configuring cues — sender vs receiver

Every cue has two sides, configured in two different places. For visual cues you keep them in sync by hand; audio cues are driven by one shared file.

The two sides

SideWhere it's setWhat it defines
ReceiverUnity — SimpleMessageReceiver, Inspector slots cue_1…cue_8What each cue number actually does (which GameObject it toggles).
SenderThe controller webpage (its own cue list)Which buttons the operator sees and what number each one sends.

The two are linked only by the cue number — there is no shared file for visual cues. Change one side and you must change the other, or they drift.

⚠️

Visual cues are limited to 8 hardcoded slots in Unity (cue_1–cue_8); adding a 9th means editing the C# script, not just the Inspector. In the current build, controller cue numbers 9–16 drive Shadow animation states (idle, spawn1, taunt, attack1, standup, deathimpact…) via a separate receiver.

Audio cues — one shared source

Audio (/audio/*) is the exception. Cues are defined in a CSV (audio_cues.csv, hosted on GitHub) that both Unity and the controller read. Edit it once — via csv-editor.html or on GitHub — and both sides update. No manual double-entry.

Publishing a new CSV, and getting every device onto it

  1. Publish: commit the edited CSV to the GitHub repo — GitHub is the single source of truth; there is no push-from-device.
  2. Operator: tap ↻ CSV to re-fetch and rebuild the buttons — or use ⟳ CSV → ALL, which reloads locally and sends /audio/reload so every device in the selected audience re-fetches the CSV and preloads any new stems, no restarts.
  3. Confirm: each performer board shows a fresh “config: ready”. Do this while devices have internet (they fetch from GitHub) — after that, everyone is cached and offline-safe.
📴

Offline resilience: the controller caches the last successfully fetched CSV and rebuilds from it if the fetch fails (shown as “offline — using cached CSV from <date>”), and performer devices fall back to their disk cache the same way. A closed show network can no longer lose the cue list to a page reload. Unity does not handle /audio/reload yet — headsets pick up a new CSV at app launch until that handler lands.

What the controller saves

The controller persists your setup in the browser (localStorage), and you can Export / Import Settings (a JSON file) to back it up or move it to another machine (e.g. the operator tablet). Saved:

  • Cue groupings, group names, and colors
  • Audiences (which headsets a cue targets) and the target IP list
  • UDP port, OSC addresses, the audio CSV URL, and the fire-forward delay
💾

localStorage is per-browser, per-machine. Use Export Settings to save a backup and Import it on other devices — otherwise a fresh machine (or cleared browser data) starts blank. Note the individual cue definitions are code-defined (visual) or CSV-defined (audio); the controller saves your organization and targeting, not the cue list itself.
