Cues

The cueing system is the heart of the show — it turns the score into timed events that move phantoms, light, and sound through the city.

Cues

Overview

A cue is a single directed moment — a phantom enters, a light shifts, a voice rises. Cues are grouped into beats, and beats are driven by the Conductor Clock in time with the music.

The vocabulary of a cue

TermMeaning
BeatThe smallest directed unit of action, identified by a stable id like II.trial.enter.
CueOne action attached to a beat — spawn, move, light, audio, retire.
TriggerWhat advances to a beat: the conductor clock, audience position, or operator action.
AnchorThe real-world point a cue's content is positioned against.
Running OrderThe full ordered list of beats that makes up a performance.

How a cue fires

The clock reaches a beat

Or an audience-position / operator trigger requests it.

The beat event is raised

Onto the shared event bus — see System Overview.

Each cue in the beat executes

Phantom Director, Spatial Audio, and lighting carry out their part, anchored to the world.

The beat resolves

The clock holds or advances to the next beat, depending on the trigger type.

Cue types

🎭

Phantom

Spawn, move, animate, or retire a virtual performer.

💡

Light

Shift key light color & intensity to set the movement's emotion.

🔊

Audio

Trigger a voice or stem at a spatial source; duck or crossfade.

⏯️

Flow

Hold, wait-for-audience, or advance the running order.

Cues

Editing & Cueing System

Cues are authored as data, not code — so designers can shape a movement without recompiling. Each movement owns a cue sheet that the runtime loads and the operator drives.

The cue sheet

Every movement has a cue sheet under Assets/Movements/<movement>/cues.json. It lists beats in order, each with the cues to run and the trigger that advances to it.

{
  "movement": "II-the-trial",
  "beats": [
    {
      "id": "II.trial.enter",
      "trigger": "audience@courthouse.steps",
      "cues": [
        { "type": "light",   "color": "joan/cyan", "fade": 2.0 },
        { "type": "phantom", "id": "judge", "anchor": "courthouse.dais" },
        { "type": "audio",   "stem": "trial/aria-1", "source": "courthouse.dais" }
      ]
    }
  ]
}

Editing workflow

Author in the Cue Editor

The in-Unity Cue Editor window reads and writes the cue sheet visually — add beats, attach cues, pick anchors from the scanned map.

Preview in the sandbox

Scrub the timeline in _Sandbox.unity to see cues fire without leaving your desk (see Running a Local Test).

Tune timing to the score

Align beat triggers to musical moments. Use elastic holds where the audience sets the pace by walking.

Commit the cue sheet

Cue sheets are plain JSON — review them in a normal pull request, no scene merge required.

✅

Because cues are data, a designer can re-time an entire movement and ship it through review without an engineer recompiling. Keep them human-readable.

The operator console

At performance time, a human operator drives the cue sheet from the Show-Control console. This is the only place non-diegetic UI is allowed — the audience never sees it.

ControlEffect
Start / ResumeBegins or continues the conductor clock from the current beat.
Jump to BeatRepositions the clock — used heavily during rehearsal.
HoldFreezes phantoms and sustains the underscore for audience pacing.
E-StopFades all content to a safe neutral state immediately.
🔒

Never edit a cue sheet live during a performance. Editing happens in the sandbox and ships through review. The console drives cues; it does not author them.
