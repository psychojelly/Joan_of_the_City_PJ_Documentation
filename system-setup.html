Setup · Path 1

Running a Local Test

The fastest way to see a scene working — a single machine, a single headset (or the in-editor simulator), no venue required.

Open the project in Unity Hub

Add the cloned folder and open with Unity 6000.0.26f1. Let packages resolve on first launch.

Load the sandbox scene

Open Assets/Scenes/_Sandbox.unity — a desk-friendly scene that simulates a single scene without leaving the building.

Play in the Editor (simulator)

Press Play with the XR Device Simulator active to walk the scene with mouse + keyboard. No hardware needed.

Or deploy to a single headset

Build & Run to your device, put on the glasses, and trigger the opening cue from the Show-Control console.

Drive it from Show-Control

Step through the cues to confirm light, particles, ravens/shadow, and audio all fire correctly — and that STOP ALL halts everything cleanly.

🧪

The sandbox uses a fixed local anchor instead of a real-world scan, so you can iterate on staging and cues without on-site tracking. Treat its layout as indicative, not final.

⚠️

The Editor lies about XR frame cost. Always confirm a scene's performance on-device before calling it done.

Setup · Path 2

Larger System Set Up

The full performance configuration: multiple synchronized headsets, a real-world anchor map, an operator running Show-Control, and audio routed for the venue.

Topology

One operator station runs Show-Control and drives the cue sequence. Each audience headset runs the client build and stays in sync over the local network. A real-world anchor map (captured during the location scout) localizes every device to the same physical stage. Target headset count is still to be set — see Open questions.

NodeRole
Operator StationRuns the show, Show-Control console, STOP ALL authority.
Audience HeadsetsXREAL glasses driven by host “brain” PC(s); render virtual content & audio against shared anchors.
Anchor MapPre-scanned spatial map of each scene's stage; shared to all devices.
Local NetworkLow-latency link carrying cue sync & shared state.

Bring-up sequence

Stage the location

Arrive on-site, confirm the scene's stage matches the scanned anchor map, and re-scan if the environment has changed (moved furniture, new construction, lighting).

Start the operator station

Launch Show-Control and confirm it loads the scene's cue sequence and that STOP ALL is wired.

Localize each headset

Deploy the client build to every headset and run the localization routine until each device resolves the shared anchors. Verify alignment against a known physical reference point.

Verify shared state

Confirm all headsets see a consistent world — content appears in the same place for everyone, and cues fire identically across devices.

Sound check

Walk the route; confirm spatial audio sources sit where they should and attenuate naturally with distance and occlusion.

Dry run the scenes

Operator advances each scene's cues; the team watches for tracking drift, anchor slippage, dropped or inconsistent cues, audio lag, and comfort issues before any audience arrives.

📍

Tracking budget: real-world venues are hostile to tracking — moving people, changing light, reflective surfaces. Anchor critical content near stable, textured surfaces, and never place a key dramatic moment where tracking is known to drift. This is why anchoring (Anchor, not Follow) is a top priority.

🛟

Always verify STOP ALL before doors open. Every operator must know how to halt and reset the show to a safe state instantly. See Cues → Editing & Cueing System.

Scene measurements to capture

Each scene needs real-world dimensions recorded on-site so content places correctly. Capture these during bring-up and file them with the scene's spec.

SceneMeasurements needed
LightPlacement measurements, radius, height.
VoicePlacement measurements, raven distances, landing locations.
ShadowSpawn zones, movement ranges, surround radius.
CommunionFinal ~12′ radius, height ranges, audience clearance.

Pre-show checklist

  • All headsets charged and localized to the shared anchor map.
  • Operator station on mains power with network confirmed stable.
  • Cue sequences loaded and verified against the latest scene builds.
  • Audio levels checked at the real performance time of day.
  • STOP ALL verified on the current build; safe route and seated/limited-mobility paths confirmed.
  • OSC controller proxy running and reaching the target (see below).
Setup · Operating

Running the OSC controller

The OSC controller is how cues are fired during a show — a web page of buttons backed by a small Python proxy. See Cues → The OSC control system for how it works; this is how to run it.

One-time setup

Install Python 3.9+

From python.org/downloads. Windows: tick “Add Python to PATH” on the first install screen.

Confirm it's installed

In a terminal: python --version and pip --version (Mac: python3 / pip3). You should see version numbers.

Install the one dependency

pip install python-osc (Mac: pip3 install python-osc).

Every time you run it

Go to the folder

cd "C:\path\to\the\osc\files" (use your real path; Mac uses the matching path).

Start the proxy

python proxy.py (Mac: python3 proxy.py). You should see OSC proxy listening on http://localhost:8765. Leave this terminal open.

Open the controller

Double-click osc-controller.html (or drag it into a browser).

Point it at the target

Set Host and Port to your OSC target (defaults 127.0.0.1 / 7000) and use the buttons.

Stopping

Close the browser tab, then click the terminal and press Ctrl+C to stop the proxy.

Troubleshooting

SymptomFix
python is not recognizedPython isn't on PATH. Reinstall and tick Add Python to PATH, then reopen the terminal.
ModuleNotFoundError: No module named 'pythonosc'Run pip install python-osc again in the same terminal.
Buttons in the page do nothingThe proxy isn't running — confirm the proxy.py terminal is open and shows the “listening” message.
