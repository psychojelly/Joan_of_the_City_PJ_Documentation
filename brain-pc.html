Brain PC & XREAL Glasses

How the headset hardware actually runs the show: the XREAL glasses display and track, while a host “brain” PC does the thinking — running the Unity build, holding the synced assets, and receiving cues.

✅

Grounded in the project. The cue, OSC, XREAL, Android-build and audio-loader details below were verified against the Unity project (joan-of-the-city-xreal); the wireless brain↔headset link is confirmed by the team. The cue fan-out, stem server URL, and PuTTY's role aren't pinned down yet and stay marked (to confirm).

Brain PC

The two halves

The show is an Android build that runs on each headset's own compute and renders locally to the XREAL glasses. The brain PC does not render — it's a control & content server that sends cues and serves assets to the headsets over a local Wi-Fi network.

🕶️

Headset (XREAL + Android compute)

Brain PC

What moves where

Four flows run through the brain PC during a show.

🖼️

Render & tracking

Each headset renders the scene itself; its glasses report head pose locally. The brain isn't in this loop — no frames cross the network.

🎬

Cues

The brain fires cues over OSC across Wi-Fi; each headset's Android build receives them and runs the moment. See OSC control system.

📦

Asset / stem pull

Headsets download audio stems (and assets) from the brain over Wi-Fi and cache them locally — fetched on demand unless pre-warmed.

🛠️

Management

Terminal / file access to the brain (and possibly headsets) to start the build, check status, or move files.

📥

Two separate paths onto a headset. The app build is manually uploaded to each headset (believed — confirm the method), while audio stems & assets are pulled over Wi-Fi at runtime and cached. Because the build is per-device and manual, every headset must be on the same version — a missed upload means one headset runs an old build and diverges from the rest.

Brain PC

What runs where

The Android build (Unity 6, URP, XREAL SDK) runs on each headset and listens for two streams of OSC cues sent from the brain over Wi-Fi. Both arrive at UDP :7000 via proxy.py.

🔊

Audio stems load via AudioStemLoader, whose default mode is RemoteWithLocalFallback — it downloads stems from a server URL over Wi-Fi and caches them, falling back to bundled files only if the download fails. Pre-warm that cache before a show to avoid first-play audio lag.

Brain PC

Open questions

What the code doesn't answer — the operational and multi-user pieces still to pin down.

  • Cue fan-out — how one controller reaches multiple headsets over Wi-Fi (broadcast, multicast, or a per-headset target list?). The single hardcoded receiver IP makes this the biggest open question.
  • Stem/asset server — what the AudioStemLoader BaseUrl points to (the brain's HTTP server?), and whether stems are pre-cached before shows.
  • How many headsets — target count, and whether each has its own compute (assumed) on the shared Wi-Fi.
  • Network — the venue Wi-Fi setup, and whether it has the headroom for several headsets pulling stems at once.
  • PuTTY's role — terminal and/or file sync to the brain/headsets (not evidenced in the repo).
  • Build deployment — the build is manually uploaded to each headset (believed). Confirm the method (USB / adb sideload / file copy) and how versions are kept matching across headsets.
  • Anchoring on XREAL — plane detection + recenter handlers exist; how anchors hold long-term ties to the anchoring priority.
