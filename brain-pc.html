Brain PC & XREAL Glasses

How the hardware actually runs the show: each headset's own Android compute runs the Unity build and drives the XREAL glasses, while the operator's “brain” device — a PC or a tablet (TBD) — fires the cues.

✅

Grounded in the project. The cue, OSC, XREAL, Android-build and audio-loader details below were verified against the Unity project (joan-of-the-city-xreal); the wireless brain↔headset link is confirmed by the team. The cue fan-out, stem server URL, and PuTTY's role aren't pinned down yet and stay marked (to confirm).

Brain PC

The two halves

The show is an Android build that runs on each headset's own compute and renders locally to the XREAL glasses. The brain PC doesn't render — it's the operator's control machine: it runs the cue controller and fires cues to the headsets over a local Wi-Fi network. Audio content is pulled by the headsets from a remote (GitHub) URL, not from the brain.

Brain PC

What moves where

Four flows run through the brain PC during a show.

🖼️

Render & tracking

Each headset renders the scene itself; its glasses report head pose locally. The brain isn't in this loop — no frames cross the network.

🎬

Cues

The brain fires cues over OSC across Wi-Fi; each headset's Android build receives them and runs the moment. See OSC control system.

☁️

Audio stem pull

Headsets download audio stems + the cue CSV from a GitHub URL over the internet and cache them locally — fetched on demand unless pre-warmed.

🛠️

Management

Terminal / file access to the brain (and possibly headsets) to start the build, check status, or move files.

📥

Two separate paths onto a headset. The Unity build is put on each headset manually, one device at a time, while audio stems are pulled from a remote GitHub URL at runtime (over the internet) and cached. Because the build is installed by hand per device, every headset must be on the same version — a missed update means one headset runs an old build and diverges from the rest.

Brain PC

What runs where

The Android build (Unity 6, URP, XREAL SDK) runs on each headset and listens for two streams of OSC cues sent from the brain over Wi-Fi. Both arrive at UDP :7000 via proxy.py.

🔊

Audio stems load via AudioStemLoader, set to Remote in the production scene — it downloads stems from a GitHub URL (raw.githubusercontent.com/dlobser/opera-audio) over the internet and caches them to disk. There is no bundled fallback in the shipped config, so a device's first-ever run needs internet; after that the disk cache makes it offline-safe. Pre-warm that cache before a show to avoid first-play audio lag.

Open questions

What the code doesn't answer — the operational and multi-user pieces still to pin down.

  • Internet vs pre-cache — stems come from GitHub, so headsets need internet at first run unless the cache is pre-warmed. Confirm the show-day practice (pre-cache vs live fetch).
  • How many headsets — target count, each with its own compute on the shared Wi-Fi.
  • Network — the venue Wi-Fi setup, and whether it has headroom for cue traffic to all headsets at once.
  • Performer tablet app — it must be a sideloaded APK (browsers can't receive OSC), but which one: the Unity project's Android build, a stripped audio-only Unity build, or a separate native app? And how is it installed/updated? (Ask the Lead Developer.)
  • PuTTY's role — terminal and/or file sync to the brain/headsets (not evidenced in the repo).
  • Anchoring on XREAL — plane detection + recenter handlers exist; how anchors hold long-term ties to the anchoring priority.
